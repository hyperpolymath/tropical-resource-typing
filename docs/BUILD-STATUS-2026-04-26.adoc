== Important — methodology lesson

The v1.0 of this document, the `STATE.a2ml` checkpoint in commit
`61ee698`, and the errata stanzas in the two handover docs all said
"6 failures across 3 theories". That number came from grepping
`***`-prefixed lines off the live `isabelle build` stdout while it ran.

The **authoritative** error inventory comes from:

----
isabelle build_log -H Error <session>
----

which reads the build heap and reports every error site Isabelle
recorded — including `Bad arguments for document antiquotation`, `Type
unification failed`, `Local statement fails to refine`, etc., which my
stdout filter dropped. Using the authoritative source the picture is:

* **16 proof-failure sites** across 4 theories (Determinants ×7,
  Kleene ×4, Matrices_Full ×4, CNO ×1).
* **10 doc-antiquotation / lexical sites** (cosmetic — block PDF
  generation, not proof checking).

Every step in this repo henceforth should consult `build_log -H Error`
before claiming a count.

warnings come from running the build inside `/var/mnt/eclipse` (a
symlinked mount) without first invalidating the export DB. They affect
only the session's cached export artefacts, not proof checking. Silence
with `-c -o export_proofs=false` or by clearing
`~/.isabelle/Isabelle2025-1/heaps/*/export.db` before the build.

=== `Tropical_Determinants` (7 proof sites)

==== D-153 + D-207 — likely root (HIGH priority)

----
*** Type unification failed: Variable 'a::{one,ord} not of sort zero
*** Type error in application: incompatible operand type
***   Operator:  π :: 'a ⇒ 'a
***   Operand:   0 :: ??'a
*** At command "have" (line 153)

*** Type unification failed: Variable 'a::{numeral,ord} not of sort zero
*** ... same shape ...
*** At command "have" (line 207)
----

Both sites use `bij π` over polymorphic `'a`. In Isabelle 2024 this
inferred `'a = nat`; in Isabelle 2025-1 the inference picks
`'a::{one,ord}` (or `'a::{numeral,ord}`), which is not of sort `zero`,
and `0` then has no type. Likely a knock-on from a HOL.Permutations or
HOL.Bij class-instance change.

**Fix candidates:**
* Annotate the `bij π` site with `(\<pi> :: nat \<Rightarrow> nat)`.
* Add `for π :: "nat \<Rightarrow> nat"` to the surrounding lemma.
* Investigate whether Isabelle 2025 release notes mention a
  Permutations / Bij signature change.

**Why this is likely the root:** sites 81, 166, 191, 262, 302 are all
downstream proofs in the same theory. If the type is wrong at 153/207
and a witness type is then exported across the file, downstream
`have`s and `show`s about that witness can fail in ways that look
unrelated.

==== D-81 — likely independent

----
*** Local statement fails to refine any pending goal
*** Failed attempt to solve goal by exported rule:
***   (f x + sum f F = f x) ⟹ sum f (insert x F) ∈ f ` insert x F
*** At command "thus" (line 81)
----

`sum_tropical_min_mem` case `(insert x F)`. The proof is structurally
correct, but `auto` is not applying `sum_eq` as a substitution.

**Fix:** `thus ?thesis using sum_eq by metis` or `by (simp add: sum_eq image_iff)`.

==== D-166, D-191, D-262, D-302 — investigate after D-153/207

Likely cascade. Triage after the type-class root is fixed.

=== `Tropical_Kleene` (4 proof sites + 7 doc sites)

==== K-237 — `proof (rule sum_mono)` fails

----
*** Failed to apply initial proof method (line 237)
*** goal: (∑l<n. A i l * trop_mat_pow n A m' l j)
***       ≤ (∑l<n. A i l * trop_mat_star n A l j)
----

`sum_mono : (⋀i. i ∈ K ⟹ f i ≤ g i) ⟹ sum f K ≤ sum g K` requires the
result type to be in `ordered_comm_monoid_add`. Tropical may need a
typeclass annotation here.

**Fix:** try `proof (intro sum_mono ballI)` or
`proof (rule sum_mono[where f="..." and g="..."])`.

==== K-403 — `obtain ... by blast` (easy mop-up)

----
*** using this:
***   ∃xs ys zs y. w = xs @ [y] @ ys @ [y] @ zs
*** goal: (⋀xs v ys zs. w = xs @ v # ys @ v # zs ⟹ thesis) ⟹ thesis
----

`not_distinct_decomp` returns `[y] @ ys @ [y]`; the obtain wants
`v # ys @ v #`. blast doesn't desugar `[y]` to `y # []`.

**Fix:** `by force` or `by (metis append_Cons append_Nil)`. **Verified working in this session's build #2** (then reverted to clean baseline).

==== K-474 — `trop_mat_id ≤ trop_mat_id + trop_mat_mul`

Reflexive bound on tropical addition. Likely needs an instance lemma.

==== K-575 — `sum f S ≤ c` from finite-bounded

Standard `sum_bounded_above` should apply; unclear why it doesn't.

=== `Tropical_Matrices_Full` (4 proof sites + 2 doc sites)

==== M-1525 — root cause (HIGH priority — explains M-838 + M-1874 + M-2044)

----
*** exception THM 0 raised (line 308 of "drule.ML"):
***   OF: multiple unifiers
***   ⟦?v ∈ set (butlast ?w); ?v ∈ set (tl ?w); ?no_pos_cycle ?n ?A;
***    ?w ∈ walks ?n ?k ?i ?j⟧
***   ⟹ ∃w'. path_weight ?A ?w ≤ path_weight ?A w' ∧
***           length w' < length ?w ∧ w' ≠ [] ∧
***           hd w' = hd ?w ∧ last w' = last ?w ∧ set w' ⊆ {..<?n}
*** At command "using" (line 1525)
----

`path_weight_cycle_excise` takes `v ∈ butlast w` AND `v ∈ tl w`.
Those don't pin two distinct positions for `v`: a single occurrence
at position `p` with `0 < p < length w − 1` satisfies both. OF then
finds multiple unifiers because the cycle to excise is not uniquely
determined.

**Fix — option A (recommended): strengthen the lemma signature.**

----
lemma path_weight_cycle_excise:
  assumes "0 < p" "p < q" "q < length w - 1" "w ! p = w ! q"
  assumes "no_pos_cycle n A" "w ∈ walks n k i j"
  shows "∃ w'. path_weight A w ≤ path_weight A w'
              ∧ length w' < length w ∧ w' ≠ []
              ∧ hd w' = hd w ∧ last w' = last w
              ∧ set w' ⊆ {..<n}"
----

The four downstream call sites (838/1874/2044/1525) all already have
the position pair implicitly in their pigeonhole context — they need a
small witness construction.

**Fix — option B:** rewrite `path_weight_cycle_excise` body to do the
pigeonhole on pairs `(p, q) ∈ {0..<length w}²` directly. Bigger
refactor.

Effort: 30–60 min.

==== M-838 + M-1874 — pigeonhole (cascade from M-1525)

Both sites have the identical false-as-stated subgoal:

----
*** ⟦Suc p0 ≤ q; Suc p0 < length w; q < length w - Suc 0;
***   w ! q = w ! Suc p0; v = w ! Suc p0⟧
***  ⟹ Suc p0 < q
----

`Suc p0 ≤ q` does not imply `Suc p0 < q` unless `Suc p0 ≠ q`. The
"distinct positions" rule-out is missing because the surrounding
proof relies on the (broken) `path_weight_cycle_excise` interface.
Falls out of M-1525 fix.

==== M-2044 — walks set membership (cascade from M-1525)

Same root: needs the walks-set unfolding that M-1525's strengthened
interface gives.

=== `Tropical_CNO` (1 proof site)

==== C-655 — `Bad context for command qed`

Current shape (lines 619-656):

----
proof clarify
  fix i k w assume hi: "i < n" and hw: "w ∈ walks n k i i"
  show "path_weight ... ≤ 1"
  proof -
    ...
  qed
qed
----

This session attempted three rewrites (`proof (intro allI impI ballI)`,
`proof (intro allI impI allI ballI)`, `proof safe`). All produced the
same Bad-context error. The hidden problem is that **no_pos_cycle_def
unfolds to four nested binders** (`∀i. i<n ⟶ ∀k. ∀w∈S. P`) and the
inner `proof - ... qed` block fights the outer block-counting.

**Fix to try:** flat style — drop the inner `proof -`:

----
unfolding no_pos_cycle_def
proof (intro allI impI allI ballI)
  fix i k w assume hi: "i < n" and hw: "w ∈ walks n k i i"
  have step1: "path_weight (trop_mat_star n A) w \<le> trop_mat_star n A i i"
    using path_weight_star_le[OF assms(1) hi hi hw] .
  have step2: "trop_mat_star n A i i \<le> 1"
    by ⟨...inline body...⟩
  show "path_weight (trop_mat_star n A) w \<le> 1"
    using le_trans[OF step1 step2] .
qed
----

Effort: 15–20 min careful Isar work.
