== Critical bugs surfaced this session

. **`run_probe` parser scope** (`crates/burrower-core/src/attempt.rs`)
  — original code matched success/failure markers in `stdout` only.
  Echidna's `OutputFormatter` writes `✗ Proof verification failed` to
  STDERR, so the swarm couldn't tell real failures from inconclusive
  output. **Fixed in this session** — parser now scans both streams +
  falls back on exit code.

. **STATE.a2ml stale within the same day** (process bug, not code).
  The morning STATE referenced `Tropical_Kleene.thy:403` as K-403; by
  afternoon that line had moved during proof-repair edits. ADR-002
  amended in `META.a2ml`: re-run `isabelle build_log -H Error` at
  session-START, not just session-close.

. **`generate_probe` always uses `imports Main`** (known limitation,
  unfixed). Burrower's probe wrapper hardcodes `imports Main`, so even
  with EI-1 the swarm cannot probe goals referencing project theories
  like `walk_concat`. EI-1 helps echidna directly; the matching change
  to `generate_probe` (per-goal imports list) is the obvious follow-up.

. **`SocketGuard` doesn't fire on SIGTERM** (known limitation). Rust's
  Drop only runs on normal teardown. The bind-side `remove_file` is
  defensive enough to handle stale sockets, so this is informational.
  A signal handler would be a small future improvement.

== Honest swarm-load-bearing assessment

**Was the swarm load-bearing on the headline goal?**

* **For routing and observation**: YES. Three specialists engaged on
  the probe, each ran its playbook, the ledger captured 12 structured
  entries with timestamps, tactic scripts, and failure classifications.
  Without the swarm, this session would have produced no machine-
  readable record of what tactics were tried.

* **For fix discovery**: NO. Zero proofs closed. The parser bug
  meant every attempt was misclassified as `generic-failure` instead of
  the real underlying issue (`Unable to figure out induct rule` for the
  Combinatorialist's `induction xs` attempt, etc.). With the parser
  fixed in-session, the next swarm run will produce actionable
  anti-patterns, but THIS session's attempts did not.

* **For the meta-question — does Burrower's specialist abstraction
  match what proof repair actually needs?** Mixed.
+
The 4-tactic playbooks per specialist are too small to cover the real
failure modes seen in the inventory. Real fixes I've personally
applied to this repo over the past few weeks involved tactics like:
+
- `by (subst foo) (auto simp: bar_def)` — substitution + simp
- `by (cases w) (auto simp: walk_def)` — case-split + simp
- `by (metis append_Cons append_Nil)` — metis with specific lemma names
- `unfolding foo_def using h1 h2 by simp` — explicit unfold + use
+
None of those shapes are in any current specialist playbook. The
swarm's value will become real once playbooks are extended (or once
ECHIDNA's `suggest` verb lands per EI-2 and the swarm composes
playbooks from ECHIDNA's suggestions).

* **Overall verdict**: **the dogfood was foundation work, not proof
  work**. The session made the swarm USABLE for the next dogfood; it
  did not USE the swarm to repair proofs. That is still a successful
  load-bearing test for "the swarm leaving real artefacts" — the
  artefacts exist and are in `/tmp/burrow-swarm-headline-2026-04-26.jsonl`.

**What needs to change before the next dogfood**:

. Apply the `run_probe` parser fix (already in this session's commit).
. Extend `generate_probe` with a per-goal imports list so probes can
  reference project theories.
. Extend specialist playbooks with the substitution / case-split /
  metis-with-names patterns that real proof repair uses.
. Run `burrower swarm` (corpus-based) BEFORE `burrower attempt` so
  consensus-home signals enter the ledger.
. Index a corpus of the project's existing `.thy` files so the
  swarm has lemma neighbours to recommend.
