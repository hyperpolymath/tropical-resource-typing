== What this is, and how it is designed to work

*What it is.* A short, dated, jointly-signed snapshot of what we can _honestly
and verifiably_ claim about this repo at one exact commit. Nothing here is
marketing and nothing is a promise about the future — both of those live in the
README. This file is the receipt.

*How it is designed to work.* Three moving parts make it trustworthy:

. *Ground truth, not memory.* Every claim below was produced by _running the
  tool in the session that wrote this file_ — `lake build`, a clean rebuild from
  source, and a `#print axioms` audit. Where a status doc or the repo's own
  README/EXPLAINME said something the live run contradicts, the live run wins
  and we flag the contradiction (see <<Companion documents and repo metadata (cross-check)>>).
. *A frozen anchor.* The file names the exact commit SHA, branch, UTC timestamp,
  and toolchain (see <<Verifiable anchor>>), so "true" always means "true _at
  this point_". Move the SHA and the file becomes a draft until re-run.
. *A real signature.* It is landed by a *signed git commit*; that signature over
  this content at the anchored SHA is what makes the affirmation tamper-evident
  and attributable — not the prose alone.

*We are fallible.* We can be wrong, stale, or simply mistaken about our own
work. This file is our best honest belief, not a proof of its own correctness.
Treat it as a falsifiable claim, not gospel.

== The epistemic contract (read this before you trust _or_ attack)

This document records our *best joint belief* at the timestamp below. It is
*not a guarantee of correctness.* We may be wrong. We may be deluded. We may
have missed something — a hole, a stale doc, a proof that checks an idealised
core we then fail to match in a wider implementation.

*No intentional overclaim.* That is the only guarantee here: we have not
_knowingly_ inflated anything. Where something is checked-by-the-Lean-kernel we
say *proved*; where it is merely written in prose, tested, or argued, we say so;
where a checker could not run in this environment (Isabelle), we say so and do
*not* let it ride on the Lean result. If you find an honest claim here that
turns out false, that is an error to be fixed — not a lie.

*Standing invitation to refute.* You are _invited_ to bulldoze any claim in this
file. We want that. But the fair form of the attack is:

. Read this file at the stamped commit.
. Reproduce (or fail to reproduce) the checks in <<reproduce,Reproduce it yourself>>.
. _Then_ tell us where the discrepancy is, against the artefact as it stood at
  this moment — so we can either justify it or concede it on the record.

An attack that skips steps 1–2 is attacking a strawman of a different date.

*No fights before the facts are cleared up.* We are not interested in a dispute
over a claim until the discrepancy has actually been checked against the
artefact at this commit and we have had the chance to either justify it or
concede it on the record. Good faith both ways: bring a reproducible
discrepancy and we will fix the claim, the doc, or the code — promptly and
without defensiveness. The goal is a corrected record, not a won argument.

=== The honest nuance you must not lose

The Lean proofs establish soundness/refinement properties of *idealised core
calculi* — speculative session-type grading, an adapter-path bottleneck
semiring, and an abstract `ResourceAlgebra` interface. They do *not* prove that
any production resource-typing *implementation* refines those calculi. There is
no executable checker in this repo wired to these proofs: `impl/` is a single,
unbuilt `TropicalDeterminant.jl`, and the `tests/` harness validates the PDF
artefact's structure, not the calculus. "Proved core" + "tested / unbuilt
implementation" is exactly that — two things, with an unverified (indeed largely
absent) bridge between them. Anyone reading "the proofs are green" as
"tropical-resource-typing is a proven-sound type *system in use*" is
overclaiming. What is proved is the *mathematics of the grading*, green and
axiom-clean; what is not proved is any compiler or runtime that consumes it.

== Reproduce it yourself

From the repo root, at the commit in <<Verifiable anchor>>, with Lean 4.13.0
(elan will fetch it from `lean-toolchain`):

[source,sh]
----
# 1. Clean, from-source build of the Lean library (no Mathlib, no network deps):
lake clean
lake build
# expect: exit 0, "Build completed successfully.", 20 steps green.
# One informational line is expected and is NOT an error:
#   info: TropicalSessionTypes.lean:237:0: inferInstance : CommSemiring Tropical

# 2. Confirm no cheating in the Lean sources (expect only a comment hit):
grep -rn 'sorry\|admit\b\|^axiom\|native_decide' \
  Resource.lean Resource/ TropicalSessionTypes.lean TropicalAdapterPath.lean

# 3. Axiom audit of the headline theorems:
cat > _axiom_audit_tmp.lean <<'EOF'
import TropicalSessionTypes
import TropicalAdapterPath
import Resource
#print axioms Hyperpolymath.Tropical.soundness
#print axioms Hyperpolymath.Tropical.tropical_grade_le_sequentialTotal
#print axioms Hyperpolymath.ProtocolSquisher.Tropical.hub_ceiling
#print axioms Resource.parametric_resource_transport
EOF
lake env lean _axiom_audit_tmp.lean
rm -f _axiom_audit_tmp.lean
# expect exactly:
#   'soundness' depends on axioms: [propext]
#   'tropical_grade_le_sequentialTotal' depends on axioms: [propext, Quot.sound]
#   'hub_ceiling' depends on axioms: [propext]
#   'parametric_resource_transport' does not depend on any axioms
----

The Isabelle layer is *not* part of this green claim. If you have Isabelle 2025-1
installed you may run `just isabelle-build` (= `just test`), but per
`docs/BUILD-STATUS-2026-04-26.adoc` expect it to be *red* (16 proof-failure
sites); it was *not* run in the session that wrote this file.
